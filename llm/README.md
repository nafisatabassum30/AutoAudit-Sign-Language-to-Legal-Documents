# Bangla-FIR LLM

This is the **LLM stage** of the AutoAudit pipeline: it takes the raw
Bangla text produced by the upstream BdSL sign-recognition stage
(MediaPipe + ST-GNN) and converts it into a structured, FIR-ready
legal complaint.

```
"আমার মানিব্যাগ চুরি হয়েছে উত্তরা বিকাল ৫টায়"
                    │
                    ▼
          [ fine-tuned LLM (this module) ]
                    │
                    ▼
   {
     "date_of_occurrence": "...",
     "time_of_occurrence": "বিকাল ৫টায়",
     "location": "উত্তরা, বাংলাদেশ",
     "offense_type": "চুরি",
     "complainant_name": "...",
     "complainant_address": "...",
     "complainant_contact": "...",
     "accused": "অজ্ঞাত",
     "description": "<formal legal paragraph>"
   }
                    │
                    ▼
        formatted, submission-ready FIR document
```

## Contents

```

├── data/
│   ├── schema.md                # dataset format spec
│   ├── seed_dataset.jsonl        # synthetic bootstrap dataset (see caveat below)
│   ├── raw/                      # put real, collected data here
│   └── processed/                # generated by src/data_preparation.py (gitignored)
├── scripts/
│   ├── generate_seed_dataset.py  # synthesizes the bootstrap dataset
│   └── smoke_test.sh             # end-to-end pipeline smoke test (CPU, tiny model)
├── src/
│   ├── fir_schema.py             # FIR JSON schema, parsing, validation, rendering
│   ├── prompts.py                # prompt/chat-template construction
│   ├── data_preparation.py       # raw JSONL -> tokenized, loss-masked training data
│   ├── train.py                  # LoRA / QLoRA fine-tuning entrypoint
│   ├── infer.py                  # load adapter + generate a FIR for new input text
│   ├── evaluate.py                # field-level accuracy + ROUGE-L on a validation split
│   └── api.py                    # Flask API wrapping inference for the rest of the pipeline
├── tests/                        # unit tests (pytest)
└── requirements.txt
```

## Important caveat about the data

`data/seed_dataset.jsonl` is a **synthetic, template-generated
bootstrap dataset** (see `scripts/generate_seed_dataset.py`). It exists
so that this entire pipeline -- data prep, training, inference,
evaluation, the API -- could be built and verified end-to-end without
waiting on the (still-in-progress) real dataset. It is **not**
sufficient to produce a production-quality model by itself: it has
limited vocabulary/phrasing diversity and made-up FIR text.

Per the project's data-collection plan, you still need to:

1. Collect 1,000+ real (anonymized) FIR templates/examples (e.g. via
   BLAST or police records).
2. Pair each with either (a) a real BdSL-derived sentence (once the
   ST-GNN stage is trained and can process real sign videos) or (b) a
   native-speaker-written approximation of what a signer would convey.
3. Add these to `data/raw/` in the same JSONL format described in
   `data/schema.md`, then fine-tune on the combined (or replaced)
   dataset.

## Setup

```bash
cd llm
pip install -r requirements.txt
```

- `bitsandbytes` (needed for 4-bit QLoRA) only works with a CUDA GPU.
  On CPU-only machines, training automatically falls back to plain
  LoRA (see `--no-4bit` below) -- useful for development/testing, not
  for producing a usable model.
- To use a gated base model (e.g. `meta-llama/Meta-Llama-3.1-8B-Instruct`),
  accept its license on Hugging Face and run `huggingface-cli login`
  (or set `HF_TOKEN`) first. Ungated alternatives like
  `Qwen/Qwen2.5-7B-Instruct` need no extra setup and have strong
  multilingual (including Bangla) support out of the box.

## Usage

### 1. Prepare data only (optional -- `train.py` does this automatically)

```bash
python -m src.data_preparation \
  --raw data/seed_dataset.jsonl \
  --out data/processed \
  --tokenizer Qwen/Qwen2.5-7B-Instruct
```

### 2. Fine-tune (LoRA / QLoRA)

Real GPU training run, using the defaults in `configs/training_config.yaml`:

```bash
python -m src.train --config configs/training_config.yaml
```

Override anything from the CLI, e.g. to point at real, larger data and
a specific output directory:

```bash
python -m src.train \
  --model-name-or-path Qwen/Qwen2.5-7B-Instruct \
  --raw-data data/raw/fir_dataset.jsonl \
  --output-dir outputs/bangla-fir-lora-v1
```

CPU smoke test (tiny model, a handful of steps, no quantization --
only meant to validate the pipeline runs, not to produce a usable
model):

```bash
bash scripts/smoke_test.sh
```

### 3. Run inference

```bash
python -m src.infer \
  --base-model Qwen/Qwen2.5-7B-Instruct \
  --adapter outputs/bangla-fir-lora \
  --text "আমার মানিব্যাগ চুরি হয়েছে উত্তরা বিকাল ৫টায়"
```

Prints the FIR JSON followed by the formatted, submission-ready
document.

### 4. Evaluate

```bash
python -m src.evaluate \
  --base-model Qwen/Qwen2.5-7B-Instruct \
  --adapter outputs/bangla-fir-lora \
  --val data/processed/val.jsonl
```

Reports per-field exact-match accuracy, JSON-parse success rate, and
ROUGE-L on the free-text `description` field.

### 5. Serve as an API (for the rest of the pipeline to call)

```bash
FIR_BASE_MODEL=Qwen/Qwen2.5-7B-Instruct \
FIR_ADAPTER_PATH=outputs/bangla-fir-lora \
python -m src.api
```

```bash
curl -X POST http://localhost:5000/generate-fir \
  -H "Content-Type: application/json" \
  -d '{"text": "আমার মানিব্যাগ চুরি হয়েছে উত্তরা বিকাল ৫টায়"}'
```

## Design notes

- **Model-agnostic**: `train.py`/`infer.py` work with any Hugging Face
  causal LM (Qwen2.5, Llama-3.x, Gemma-2, or even a small model for
  local development). Swap `--model-name-or-path` / `--base-model`.
- **Loss masking**: `data_preparation.py` masks prompt tokens with
  `-100` so the model is only trained to predict the FIR JSON
  completion, not to reproduce the instructions.
- **Structured output contract**: `fir_schema.py` is the single source
  of truth for the FIR fields, used consistently by data prep,
  training, inference, and evaluation, so they can never drift apart.
- **Robust parsing**: `parse_fir_output` tolerates markdown fences and
  extra chatter around the JSON object, since instruction-tuned models
  occasionally add commentary despite being told not to.
- **Context-length safety**: `FIRGenerator` clamps `max_new_tokens` to
  whatever headroom remains in the base model's context window, so
  generation never crashes on the position-embedding limit regardless
  of the (prompt length, base model) combination.
